	Қайырлы күн, құрметті студенттер! Мен сіздерді келесі зертханалық жұмысқа шолу сессиясында көргеніме қуаныштымын. Бүгінгі зертханалық жұмыстың тақырыбы - Scale & Load Balance your Architecture. Сонымен, бастайық.
	Бізге бірінші кезекте AWS Academy-да тиісті Cloud Foundations курсын ашу керек, сосын сол жақ навигация мәзіріндегі Modules өту керек. Ашылған парақшада Module 10 – Auto Scaling and Monitoring орналасқан және сіз Lab 6 – Scale & Load Balance your Architecture белсенділігін көресіз. Негізгі зертханалық жұмыс парақшасына өту үшін сілтеме бойынша өтейік. Мұнда зертханалық жұмыс тапсырмасын оқуды жеңілдету үшін Terminal терезесіне сәйкес пункттің құсбелгісін алу арқылы жасырып, тек README қалдыра аламыз. Сондай-ақ, сессияға уақытша AWS аккаунтын құруды бастау үшін Start Lab батырмасын басуға болады.
        Бұл зертханалық жұмыста біз Elastic Load balancing - ELB және Auto Scaling сервистерін толығырақ зерттейміз. ELB-ді біздің таргеттерімізге арналған трафикті теңестіру үшін, ал Аuto Scaling-ті IT-ресурстарымызды динамикалық түрде көбейту немесе азайту үшін қолданамыз. ACG-мен, яғни Auto Scaling Group-пен бірге қолданылатын ең көп таралған сервис – бұл Amazon EC2 инстанстары.
        Бүгін біз сіздермен не істейміз? Біріншіден, біз жұмыс жасап тұрған инстанстың AMI көшірмесін жасаймыз. Әрі қарай, біз load balancer құрамыз, Auto Scaling group-ты конфигурациялаймыз және іске қосамыз, сондай-ақ осы сервистердің барлығы байланыста қалай жұмыс істейтінін, Auto Scaling қалай жүзеге асатынын көреміз және біз бұл өзгерістерді alarms-тарда көре аламыз және бұл процесті Amazon CloudWatch-те бақылай аламыз. Бұл зертханалық жұмысты орындау үшін 30 минут жұмсау ұсынылады. Естеріңізге сала кетейін, егер сізге көбірек уақыт қажет болса, сіз әрқашан  есептегішті нөлге келтіріп, зертханалық жұмысты жалғастыра аласыз.
	Зертханалық жұмысымызды бастамас бұрын, AWS ішіндегі қазіргі архитектура келесідей екенін көреміз. Осы зертханалық жұмыстың аясында не жасалады: Load Balancer құрамыз, Public Subnet 2-де орналасқан Web Instance жоғарғы оң жақтан Private Subnet-ке көшіреміз және Load Balancer мен Auto Scaling байланысын реттейміз. Осылайша, қажет болған жағдайда бізде екі Availability Zones-та инстанстар жасалынады, яғни Availability Zone A және Availability Zone B.
	Ең алдымен, зертханалық жұмысты бастау үшін біз AWS Management Console-ге жетуіміз керек. Ол үшін, егер сіз әлі жасамаған болсаңыз, жоғарғы оң жақтағы Start Lab батырмасын басуыңыз керек. Оны басқаннан кейін, зертханалық жұмыстың күйін көрсететін қалқымалы терезе пайда болады, яғни "Lab Status: in creation". Күй Ready күйіне өзгергеннен кейін, сіз осы терезені жаба аласыз, содан кейін зертханалық жұмыс сессиясы үшін жасалған уақытша AWS Account-қа өту үшін AWS батырмасын басуыңыз керек.
	AWS Management Console-ге жеткеннен кейін біз бірінші тапсырманы орындауға өте аламыз. Мұнда біз AMI көшірмесін жасауымыз керек. Бұл AMI Auto Scaling group-тың атынан жаңа инстанстар жасау үшін пайдаланылады. Мұны істеу үшін біз EC2 деп сервис іздеу жолағында теруді бастауымыз керек немесе Services батырмасын басып, ашылмалы мәзірден Amazon EC2 сервисін табу керек. Кіргеннен кейін сол жақ навигациялық мәзірде біз Instances таңдауымыз керек және белгілі бір аймақтағы AWS Account-тағы барлық инстанстардың тізімі көрсетіледі. Егер сіз инстанстарды көрмесеңіз, сізде North Virginia немесе us-east-1 аймағы тұрғанын тексеріңіз. Web Server 1 деп аталатын инстансты көруіңіз керек. Status Checks бағанында 2/2 тұрғанын тексеру керек, бұл сервер сәтті жасалғанын және жұмыс істеуге дайын екенін көрсетеді. Мұны көргеннен кейін сізге сол инстансты таңдау керек. Әрі қарай, Actions мәзір пунктінде Image and templates опциясын таңдаңыз. Әрі қарай, Create image опциясын басыңыз. Ашылған парақшада біз Image name: WebServerAMI деп көрсетуіміз керек. Әрі қарай, Image description ретінде Lab AMI for Web Server деп көрсету керек және ең төменгі жағында Create Image батырмасын басу керек. Сіз жаңа жасалған AMI және оның сәйкес ID көресіз. Сізге бұл ID-ді көшіру керек, өйткені біз оны сәл кейінірек келесі тапсырмада қолданамыз.
	Біз екінші тапсырмаға өтеміз және мұнда Load Balancer құруымыз керек. Load Balancer – бұл есептеу қуатына жүктемені теңестіруге көмектесетін IT-ресурс екенін еске саламын. Біздің жағдайда – бұл Amazon EC2 Instances және Біз Load Balancer-ді бірнеше Availability Zones-тағы трафикті теңестіретіндей етіп орнатамыз. Ең алдымен, біз Target Groups құруымыз керек, яғни бұл Load Balancer-ден трафик қайда кететінін көрсететін ресурстар. Сол жақ навигациялық мәзірден Target Groups таңдау керек. Әрі қарай, ашылған парақшада Create target group батырмасын басу керек. Target type ретінде біз Instances таңдауымыз керек, Target group name ретінде LabGroup көрсетейік, сонымен қатар VPC көрсетуіміз керек. Ашылмалы мәзірден Lab VPC таңдайық. Әрі қарай, біз Next батырмасын басуымыз керек және сіз Register targets бөлімін көресіз. Қазіргі уақытта бізде targets жасалмаған, сондықтан біз бұл қадамды өткізіп, Create target group батырмасын баса аламыз.
	Енді біз Load Balancer құруға тікелей көшеміз. Ол үшін сол жақ навигациялық мәзірден Load Balancers таңдау керек. Ашылған парақшада Create Load Balancer батырмасын басамыз, сізге жаңа Load Balancer түрін таңдау қажет болатын тағы бір парақша  ашылады. Бізді Аpplication Load Balancer қызықтырады, осы опция астындағы Create батырмасын басайық. Жаңа Load Balancer параметрлерін енгізу үшін парақша ашылады. Name ретінде LabELB көрсетейік. Әрі қарай, Network желісінің параметрлерінде VPC таңдау керек, біздің жағдайда бұл Lab VPC. Осыдан кейін біз Availability Zones таңдауымыз керек, ол үшін бірінші Availability Zone – us-east-1a таңдаймыз. Мұнда біз Public Subnet 1 таңдауымыз керек, екінші Availability Zone үшін де солай істеу керек және Public Subnet 2 таңдау керек. Нәтижесінде сізде екі Subnet таңдалуы керек, бұл Public Subnet 1 және Public Subnet 2.	
	Біз келесі бөлімге көшеміз – бұл Security groups. Мұнда ашылмалы мәзірден Web Security Group опциясын таңдау керек. Сондай-ақ, default security group таңдалғанын көре аласыз, ол бізге қажет емес, сондықтан X батырмасын басып, default security group жою керек. Нәтижесінде бізде Web Security Group деп аталатын жалғыз security group қалады. Ең соңғы параметр – Listener, яғни трафик қай портқа, қайсы протокол арқылы келетіндігі. Біздің жағдайда бұл HTTP:80. Сондай-ақ, мұнда біз Default action ретінде трафикті LabGroup-қа бағыттауымыз керек. Осымен біз load balancer үшін барлық параметрлерді енгізуді аяқтадық. Енді парақшаның төменгі жағына жылжып, Create load balancer батырмасын басу керек. Load balancer сәтті жасалғаны туралы хабарламаны көресіз. Сондай-ақ, load balancer парақшасын ашу үшін View load balancer батырмасын басуыңызға болады. Сіз қазір provisioning күйін көресіз, біраз уақыттан кейін ол ready немесе success күйіне ауысады. Осылайша ол жұмысқа дайын болады. Load balancer дайын болған сәтті күтудің қажеті жоқ, өйткені келесі тапсырма Auto Scaling сервисінде келесі IT-ресурстарын жасайды. Сондықтан біз күй өзгеруін күтпестен келесі тапсырмаға кірісе аламыз.
	Біз сіздермен үшінші тапсырмаға көшеміз. Бұл көлемді тапсырма. Мұнда біз Launch Configuration және Auto Scaling Group құрамыз. Мен сізге еске салайын, Auto Scaling Group – бұл белгілі бір метрикаға байланысты, мысалы, CPU жүктемесі, таргет санын өзгертуге мүмкіндік беретін IT-ресурс. Біздің жағдайда бұл есептеу қуаты немесе Amazon EC2 инстанстары. Осылайша, егер біздің Amazon EC2 инстанстарына жүктеме артып жатса, онда Auto Scaling Group жүктемені тарату үшін автоматты түрде жаңа инстанс жасайды. Егер жүктеме минималды болса, онда Auto Scaling Group те бізде Amazon EC2 инстанстарының артық саны бар екенін көреді, содан кейін артық Amazon EC2 инстанстары автоматты түрде өшеді. Сонымен, бірінші бізге Launch Configuration жасау керек. Ол үшін сол жақ навигация мәзірінде Launch Configurations опциясын таңдау керек. Ашылған парақшада Create launch configuration батырмасын басып, осы IT-ресурс бойынша деректерді енгізуді бастаймыз. Name ретінде біз LabConfig көрсетеміз. Әрі қарай, AMI ретінде біз бұрын жасаған AMI-ді таңдаймыз - бұл Web Server AMI. Instance type ретінде t3.micro таңдаймыз. Егер бұл опция қол жетімді болмаса, онда біз t2.micro таңдай аламыз.
	Келесі Additional configuration бөлімінде және Monitoring бөлімінің астында Enable EC2 instance detailed monitoring within CloudWatch жанына құсбелгі қою керек. Бұл Auto Scaling біздің инстанстарымызға жүктеменің өзгеруіне тезірек әрекет етуі үшін қажет. Келесі бөлім – Security groups. Мұнда біз Select an existing security group опциясын таңдауымыз керек, яғни бұрын құрылған Security group және ашылмалы мәзірден Web Security Group таңдаймыз. Әрі қарай жылжиық. Key pair бөлімі - бұл SSH арқылы Amazon EC2 инстанстарына қосылуға мүмкіндік беретін жеке кілт. Мұнда біз Choose an existing key pair опциясын таңдап, ашылмалы тізімнен key pair: vockey таңдаймыз. Сондай-ақ, осы кілттің Amazon EC2 инстанстарына бекітілгенін растайтын құсбелгіні қойыңыз және Create launch configuration батырмасын басыңыз. Launch Configuration құруды аяқтағаннан кейін біз оның негізінде Auto Scaling group құра аламыз. Мұнда сіз дұрыс аймақта екеніңізді тексеруіңіз керек, дұрыс аймақ – North Virginia немесе us-east-1. Бізге тізімнен LabConfig таңдап, Actions батырмасын басуымыз керек, ашылмалы мәзірден Create Auto Scaling group опциясын таңдаймыз. Сіз оны басқаннан кейін сізді Auto Scaling group құру парақшасына бағыттайды. Бұл парақшада Name өрісіне Lab Auto Scaling Group деп теріп, Next батырмасын басамыз. Әрі қарай, желі параметрлерінде біз VPC таңдауымыз керек, біздің жағдайда бұл Lab VPC. Егер "No public IP address" хабарламасын көрсеңіз, бұл хабарды елемеуге болады. Төменде Subnets таңдау керек жерде Private Subnet 1 және Private Subnet 2 таңдау керек. Неліктен Public Subnet емес, ал Private Subnet? Себебі, бұрын бізде саны, көлемі өспейтін бір ғана Amazon EC2 инстансы болды және бұл статикалық инстанс болды. Егер жүктеме көбейсе, инстанс бұл жүктемеге төтеп бере алмай ажыратылатын еді. Қазір біз жүктемеге байланысты есептеу қабілетін динамикалық түрде өзгерту мүмкіндігін қосамыз, бұл жағдайда трафикті қабылдайтын ресурс Load Balancer және Auto Scaling group болады, яғни олар қазірдің өзінде Public Subnet ішінде болады. Осылайша, қауіпсіздікті арттыру үшін біз барлық Load Balancer инстанстарын Private Subnet-те орналастыра аламыз. Subnets таңдағаннан кейін, Next батырмасын басамыз және Load balancing – optional ашылған парақшада Attach to an existing load balancer таңдау керек. Біз оны басқаннан кейін ашылмалы мәзір пайда болады, біз LabGroup таңдауымыз керек. Additional settings бөлімінде опцияны таңдап, Enable group metrics collection within CloudWatch құсбелгіні қою керек. Бұл метрикалар әр минут сайын қабылдануы үшін қажет және бұл, өз кезегінде, біздің Auto Scaling group тезірек жауап беруге мүмкіндік береді. Next батырмасын басамыз. Мұнда біз Group өлшемдерін таңдауымыз керек. Үш параметр бар: Desired capacity, Minimum capacity және Maximum capacity. Desired capacity - бұл Auto Scaling group құрудың басында құрылатын таргеттер саны. Minimum capacity және Maximum capacity, сәйкесінше, Auto Scaling group жасай алатын таргеттердің минималды және максималды саны болып табылады. Бұл жерде осындай сәт, сіз компанияның қаражатын тым көп пайдаланбау үшін минимумға және  максимумға қандай дұрыс мәнді қою керектігін түсінуіңіз керек. Сіз Maximum capacity:6 деп көрсеткеніңізді елестетіңіз және жүктеме кенеттен өсті және тіпті 6 инстанс барлық пайдаланушыларыңыздың сұрауларын өңдей алмады. Бұл жағдайда ағымдағы жүктемені өңдеу үшін 6 инстанстың жеткіліксіз екендігі туралы қосымша хабарландыруларды орнату ұсынылады, сондықтан сіз қосылып, осы мәнді қолмен өзгертуіңіз керек. Немесе бастапқыда жүктеме ұлғайған сайын біз пайдаланушылардың барлық сұрауларына қызмет көрсетуге басымдық береміз, бұл ретте инстанстар санын 10-20 инстансқа дейін көбейтуге болады деп басшылықтан рұқсат алу керек. Бұл қымбатқа түседі, бірақ сіз барлық пайдаланушыларыңызға сіздің қосымшаңыздың сәйкес әрекет ететініне кепілдік аласыз. Әрі қарай жүрейік. Scaling policies бөлімінде біз Target tracking scaling policy таңдап, оны реттеуіміз керек: Lab policy name ретінде LabScalingPolicy, Metric type ретінде Average CPU Utilization таңдаймыз, яғни біз жүктемеге, процессорларымызға қараймыз және target value ретінде 60 көрсетеміз. Біз 60 көрсеткен кезде, бұл Auto Scaling біздің барлық таргеттерімізге жүктеме шамамен 60% болуын қамтамасыз етуге тырысады дегенді білдіреді. Егер ол көбейсе, бұл жүктеме өсті және біз қосымша таргеттерді қосуымыз керек дегенді білдіреді. Егер жүктеме шамамен 40 пайыз немесе одан аз болса, онда Auto Scaling group ағымдағы таргеттер жүктемені оңай жеңе алатындығын және бір немесе бірнеше таргеттерді жоя алатындығын түсінеді.
	Біз қазір осы тапсырманың соңғы бөлігіне көшеміз. Next батырмасын басу керек. Мұнда біз тегтерді көрсетуіміз керек. Бір тег қосайық, Key ретінде Name, Value ретінде Lab Instance көрсетейік. Next батырмасын басамыз, сонда біз енгізген барлық параметрлерді қарап шығуға болатын парақша ашылады. Егер бәрі дұрыс болса, біз ең төменгі жағындағы Create Auto Scaling Group батырмасын басуға болады. Егер сіз Failed to create Auto Scaling Group қатесін көрсеңіз, пайда болған Retry Failed Tasks батырмасын басу керек, содан кейін Auto Scaling group сәтті құрылуы керек. Ең басында инстанстар саны нөлге тең болатынын және уақыт өте келе Desired capacity-ге тең мөлшерде жаңа инстанстар бапталатынын ескеріңіз, яғни соңында бізде екі инстанс бапталып, жұмысқа дайын болады.
	Біз төртінші тапсырмаға көшеміз және мұнда біз бұрын жасаған барлық IT-ресурстардың дұрыс жұмыс істеп жатқанын тексереміз. Ең алдымен, сол жақ навигациялық мәзірден Instances пунктіне өту керек. Мұнда Сіз Lab Instance деп аталатын қосымша екі жаңа инстанстың пайда болатынын көресіз. Неліктен Lab Instance? Auto Scaling group құрған кезде біз оған Lab Instance мәні бар Name тегін бердік. Осылайша, Auto Scaling group өзінен құрылған барлық инстанстар үшін өз тегтерін тапсырды. Мұнда осы екі Lab Instance үшін Health Checks өткенін күту керек. Бұл дайын болғаннан кейін, сол жақ навигация мәзірінен келесі қойындыға өтуге болады – бұл Target Groups. Тізімде тек бір target group бар екенін көресіз – бұл LabGroup. Оны басайық, сонда сіз біздің екі инстансымызды Targets қойындысында көресіз. Мұнда біз екі инстанстың да күйі Healthy күйіне өтуін күтуіміз керек. Ол үшін мезгіл-мезгіл Refresh батырмасын басуға болады. Олар Healthy күйіне өткеннен кейін, бұл олардың қазір Load Balancer сұрауларын қабылдауға қол жетімді екенін көрсетеді және сәйкесінше Load Balancer трафикті осы таргеттерге бағыттауға дайын. Енді Auto Scaling group және Load Balancer тұрғысынан бәрі жақсы екенін көрейік. Сол жақ навигация мәзірінде Load Balancers батырмасын басып, төменгі жағында Load Balancer-дің DNS Name көшіру керек. Оны көшіргеннен кейін веб-браузердегі жаңа қойындыға қойып, осы парақшаны ашуға болады. Осы зертханалық жұмыс үшін арнайы дайындалған қарапайым қосымша ашылады және сіз қандай да бір жауапты көресіз. Бұл біз Load Balancer сұрауын жібергенімізді, Load Balancer трафикті Auto Scaling group жасаған инстанстарға бағыттағанын, инстанстар бұл сұрауды өңдеп, қандай да бір жауапты қайтарғанын көрсетеді. Бұл жауапты сіз қосымшаны көрсететін парақша ретінде көресіз. Осылайша, біз Auto Scaling group және Load Balancer байланысының дұрыс жұмыс істейтінін өзімізге растадық.
	Енді біз соңғы тапсырмаға көшеміз – бұл біздің Auto Scaling-ті тестілеу, яғни инстанстарымызға жүктеме ұлғайған кезде, бұл жүктемені көбірек таргеттерге тарату үшін Auto Scaling жаңа инстанстарды автоматты түрде қосатынын тексеруіміз керек. Мұны істеу үшін біз басқа сервиске ауысуымыз керек, бұл - CloudWatch. Сервис іздеу жолағында CloudWatch теруді бастайық немесе Watch деп теруді бастауға болады, негізінен толық мәтінді іздеу жұмыс істейді. Сондықтан, ол сервис атауындағы таңбалардың кез-келген тіркесін іздейді. Сол жақ навигация мәзіріндегі CloudWatch сервисі парақшасына өткеннен кейін All alarms опциясын таңдау керек. Сізге екі alarms көрсетілуі керек, олардың екеуі де Auto Scaling group автоматты түрде жасалған. Бұл екі alarms Amazon EC2 инстанстарындағы жүктеме негізінде Auto Scaling group-ты кезекпен іске қосады, бұл өз кезегінде Auto Scaling group-та жаңа инстанстарды жоюды немесе құруды бастайды. Егер бұл alarms көрсетілмесе, сіз дұрыс аймақта отырғаныңызға назар аударыңыз. Бізге North Virginia немесе us-east-1 аймағы қажет. Егер сіз дұрыс аймақта отырып, бұл alarms көрсетілмесе, біз келесі әрекеттерді орындауымыз керек. Бұл әрекеттер тек уақытша құрылған AWS аккаунтында жасалуы керек. Нақты AWS аккаунтында мұндай жағдайлар болмайды. Сонымен, егер alarms көрсетілмесе, сізге Amazon EC2 сервисіне өту керек, содан кейін сол жақ навигациялық мәзірде Auto Scaling Groups опциясын таңдаңыз, ол біз бұрын жасаған Auto Scaling group - Lab Auto Scaling Group деп аталады. Әрі қарай, біз оны таңдағаннан кейін экранның төменгі жағында бірнеше қойынды пайда болады. Automatic Scaling қойындысын ашу керек. Тізімнен LabScalingPolicy таңдау керек, Actions батырмасын басып, ашылмалы мәзірден Edit опциясын таңдаймыз. Қалқымалы терезе ашылады және Target Value ретінде 50 деп көрсету керек және Update батырмасын басамыз. Бұл Auto Scaling group ішкі ресурстарын қайта құруды бастайды және осы жаңартудан кейін сіз Cloudwatch сервисінде осы alarms пайда болғанын көре аласыз. Назар аударсаңыз, бұл alarms бірден пайда болмайды және сіз кем дегенде 1-2 минут күтіп, содан кейін бетті жаңартасыз. Біздің alarms пайда болғаннан кейін, біз атауында AlarmHigh деп аталатын alarm таңдауымыз керек. Біз оның күйі OK екенін көреміз. Егер OK көрсетілмесе, онда біраз уақыт күтіп, refresh батырмасын басу керек. Ол OK күйіне оралуы керек. OK біздің CPU инстанстарымызға жүктеме 60%-дан аз екенін білдіреді. Ол ұлғайып, 60%-дан асқаннан кейін, бұл alarm ОК күйінен Alarm күйіне ауысады және тиісті ақпаратты Auto Scaling group-қа жібереді. Енді біз Auto Scaling group шынымен дұрыс жұмыс істейтінін тексеруіміз керек. Ол үшін біз бұрын веб-браузерде бөлек қойындыда ашылған қосымшамызға оралып, Load Test батырмасын басуымыз керек. Сіз оны басқаннан кейін біздің инстанстарға жүктеме беретін скрипттер іске қосылады. Бұл парақшаны жабуға болмайтынын ескеріңіз. Егер сіз осы парақшаны жапсаңыз, жүктеме тоқтайды. Метрикалар AWS-ке өтуін және тиісті ресурстарға хабарлануын біраз уақыт күтуіміз керек. Refresh батырмасын мезгіл-мезгіл басуға болады және 5 минуттан аз уақыт ішінде AlarmLow өзінің мәнін OK мәніне өзгертуі керек, ал AlarmHigh, керісінше, өзінің күйін In alarm күйіне өзгертуі керек, яғни қазір біздің инстанстарымызға үлкен жүктеме бар екенін айтады. Сондай-ақ, егер сіз тиісті alarm ашсаңыз, біздің жағдайда AlarmHigh, сіз график түріндегі CPU жүктемесін көресіз. Ол 60% немесе 50%-ға өткеннен кейін, егер біз alarms қайта жасасақ, және ол 3 минуттан артық созылса, ол Auto Scaling group-ты іске қосады, бұл өз кезегінде жаңа инстанстарды қосады. Біз AlarmHigh In alarm күйіне өтуін күтуіміз керек. Ол онда өткеннен кейін сіз Amazon EC2 сервисіне өтіп, сол жақ навигация мәзірінде Instances опциясын ашып, қазіргі уақытта Lab Instance атты инстанстар екі, үш немесе төрттен көп екеніне көз жеткізе аламыз. Бұл Auto Scaling дұрыс жұмыс істеп тұрғанын және жүктемені тарату үшін қосымша инстанстар жасағанын көрсетеді. Бұған көз жеткізгеннен кейін, жүктеме қалыпты күйге оралуы үшін қосымша парақшасын жабуға болады. Сондай-ақ, инстанстардың саны үш пен төрттен ең төменгі 2 мәніне қайта оралуы үшін кем дегенде 5 минут күте аласыз. Сіз бұл күйге жеткен соң, мен қосымша эксперимент жасауды ұсынғым келеді. Сізге Amazon EC2 сервисінде, Instances-тің сол жақ навигациялық мәзірінде Lab Instance деп аталатын инстанстардың бірін таңдап, Actions батырмасы арқылы бұл инстансты жою, яғни өшіру керек. Сіз оны өшіргеннен кейін, Load Balancer-де таргеттердің бірі unhealthy болғанын көресіз, содан кейін трафик осы инстансқа бағытталуды тоқтатады. Auto Scaling сәйкесінше инстанстардың саны минималды мәннен аз болғанын көреді және жаңа инстансты құруды бастайды. Біраз уақыттан кейін сіз тағы бір қосымша инстанс жасалғанын және екі инстанстың көрсетілетінін көресіз.
	Біз соңғы тапсырмаға көшеміз. Біз ең басында Web Server 1 деп аталатын инстанстан AMI құрдық. Көріп отырғаныңыздай, бізге бұл сервер енді қажет емес, өйткені қазір трафик осы Web server-ге емес, Load Balancer-ге жіберіледі. Осылайша біз бұл серверді жоя аламыз. Ол үшін инстанстар тізімінде Web Server 1 таңдау керек, содан кейін Instance state батырмасын басып, Terminate instance опциясын таңдаңыз. Сондай-ақ, қалқымалы мәзірде біз бұл инстансты өшіретінімізді растауымыз керек.
	Осымен біз зертханалық жұмыстың соңына да жеттік. Сізге уақытша AWS акаунтынан дұрыс өшіру/шығу қажет екенін еске саламын. Сондай-ақ, AWS Academy-де зертханалық жұмыс парақшасында End Lab батырмасын басып, "DELETE has been initiated... You may close this message box now." хабарламасын күтіп растау керек, бұл барлық жасалған ресурстар мен аккаунттарды жою процесі басталғанын көрсетеді және сіз бұл терезені жаба аласыз. Сіз бұл қалқымалы терезені жабасыз, сонымен қатар AWS Academy қойындысын жабуға болады және сіз зертханалық жұмысты сәтті аяқтадыңыз.
	Зертханалық жұмыстың аяқталуымен тағы да құттықтаймын. Көптеген сұрақтарға жауап берілді деп үміттенемін, сіз ELB және Auto Scaling сервистері туралы толық түсінік алдыңыз деп ойлаймын. Егер сұрақтар туындаса, сұраңыз, ал біз келесі белсенділіктерімізде көрісеміз.
